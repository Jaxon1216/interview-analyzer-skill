# STAR、追问与口播要求

## 口播版（面经中的答题正文）

- **第一人称**：面经文档里「主问题 / 追问」的**口播正文**一律用「我」作主语（协作题可一句内用「我们」说明边界）。
- **字数**：每一道**主问题**、每一道**追问**的口播版均 **≥150 个汉字**（或等价符号不计入时以汉字为主），且须能单独讲清**完整 STAR**（情境—任务—行动—结果），禁止只列关键词。
- **结构**：推荐「场景（现象）→ 归因（根因）→ 动作（分层方案）→ 结果/兜底/待测」；详见 [口播与简历抽象](oral-and-resume-patterns.md)。
- **术语**：关键 API、库名、协议可加英文括号，如：`JSON.parse`、`safeParse`。
- **待测数据**：口播中涉及性能、错误率等若暂无实测，用 **（待测）** 标出；在该文件末尾「量化与验证」中写清如何测（见 SKILL 输出契约）。

## STAR 四要素（口播内需隐含或显式点到）

- **S**：业务与现场约束（上线初期、某环境、流量特征）。
- **T**：你的目标与可验证标准（哪怕定性）。
- **A**：你具体落地的步骤与权衡（为何不用更简单的做法）。
- **R**：结果、指标或（待测）+ 后续观测/降级。

## 主问题设计

- 从**最大风险或最难权衡**切入，能自然引出架构与细节。

## 追问递进（每个亮点固定 2 个）

1. **细节**：数据结构、API、边界、错误码、一致性。
2. **失败与演化**：线上问题、回滚、债务、重做会改什么。

## 频率标注

- 题目排序：**岗位核心栈 > 跨栈基础 > 拓展**；标签 `通用` / `前端` / `后端` / `AI`。
