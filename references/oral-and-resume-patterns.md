# 口播叙述与简历一句话（抽象模板）

## 口播长答（工程向）推荐结构

来自典型高质量回答的抽象，**不固定标题名**，但信息块应齐全，便于第一人称口述 ≥150 字。

1. **场景（现象）**：线上/联调/上线初期的具体现象；可带比例、范围、如何发现（DevTools、日志、监控）。术语可加英文（如 `option`、`JSON.parse`）。
2. **归因（根因）**：为何会发生——错把不确定输出当稳定 API、缺防御层、契约不清等；一句话点出**本质**。
3. **动作（方案）**：分层写清：我先……再……最后……；可用编号列表（容错提取 → Schema 校验 → sanitize → 边界兜底）；每层说**手段 + 目的**。
4. **结果与兜底**：指标或（待测）；若仍有异常路径，写降级（如 Error Boundary）与观测。

口播语气：**面经全文第一人称「我」**；导学讲解可用「你应当理解」或中性说明，但涉及「如何答面试」的示范段可用第一人称口播。

## 简历「项目简介 / 一句话」写法

用 **1～2 句** 说清：**做什么 + 关键技术/形态 + 可选一句价值**，避免形容词堆砌。

**抽象公式**：

> 基于 {核心技术栈/范式} 的 {产品形态}，用户/业务侧能 {核心操作}，系统支持 {关键能力 A、B}，{可选：差异点如同步/异步、端侧等}。

**示例形态（勿照抄项目名）**：

> 基于 AI 大模型的智能 BI：用户上传 Excel 后，AI 自动生成 ECharts 图表与分析结论，支持同步/异步生成与拖拽看板组合多图。

面经文档中的 **项目概要** 必须接近此密度与信息量，可直接贴简历「项目描述」栏。
